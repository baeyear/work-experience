#### サービスの課題と自分が意識した事
##### 組織内でDevOpsの導入・技術的挑戦を行う体力が不足していること
- 課題
  - ビジネスサイドの要求を元に、機能開発のみをタスクとして起票していたため、以下の課題が生じていました。
    - リリースが EC2 インスタンス上の手作業によるものであり、マイグレーションやコマンドの打ち間違えでリリースに1時間程度かかる場合もありました。
    - プロダクトの初期リリース以降、 Laravel のバージョンアップがされておらず、セキュリティ脆弱性を含んだプロダクトとなっていました。
    - 外部向けのフロントエンドがMPA(Laravel blade + jQuery)であり、パフォーマンスの低下とユーザー体験の劣化が問題となっていました。
  - 加えて、上記の課題が非喫緊だが重要なタスクとして起票され、後回しになる状況が慢性化していました。

- 取り組み
  - 自分自身がアルバイトであり柔軟な立ち回りが可能だったことから、DevOps領域を自発的に改善することで開発効率の向上を試みました。
	- 以下の改善を提案した上で、主に1人で実装導入しました。
		- 開発環境のコンテナ環境改善
		- CI/CD パイプラインの導入
		- 本番環境のインスタンスを EC2 → ECS へリプレイス
		- Laravel のバージョンアップ(5.5→6.x)
		- Next.js, material UI, tailwind css, storybook の導入
    - 提案の根拠としては、特に以下を意識しました。
      - 改善に必要な実装工数と運用工数
      - チームメンバーの学習コスト、実運用までに必要な準備期間
      - 既存のインフラ基盤を有効活用できるか否か
      - プロダクション環境として実運用に耐えるか

- 結果
  - デプロイにかかる時間を1時間→10分程度に削減できました。
  - CI/CD パイプラインの実装で、静的解析と自動テストの基盤を構築しました。
    	- また、テストコードや静的解析を事前にいくつか実装することで、開発メンバーの学習ハードルを削減しました。
  - 導入したモダンフロントエンドスタックを用いてリプレイスされたサイトでは、ページインサイトのスコアが平均60点台から90点台に向上しました。
  - また、上記の取り組みが評価され、社内総会において表彰していただきました。(学生アルバイトでは唯一の表彰)
  - 上記の取り組みをきっかけに、開発組織内で DevOps に対する優先度が向上し、計画的にタスクとして起票されることが増加しました。

##### アーキテクチャの方向性が定まっていないこと
- 課題
	- 社内でアーキテクチャの共通認識が定まっていない中で開発が進行していました。
    	- MVCアーキテクチャから各自が継ぎ足して管理した結果、責務およびドメインロジックが各所に分散している状況でした。
	- ドメインロジックを体系的に記載したドキュメントや図表が存在しないため、ビジネスサイドとの間で認識のズレが発生し、業務効率が低下していました。
    	- 特に、ビジネスサイドからの問い合わせ対応に工数がかかり、開発リソースを圧迫していました。

- 取り組み
	- アーキテクチャの改善を実装例と共に提案しました。また、改善に必要なリファクタリング作業を洗い出し、開発を促進しました。
    	- 業務時間の都合上マネジメントすることは不可能であったため、他メンバーと調整して作業のマネジメントを依頼しました。
  	- 提案内容およびその根拠は以下の通りです。
    	- EloquentやFormRequestなどの、Laravelの機能を有効に活用できるか
    	- ドメインロジックを十分に分離できるか
    	- 各層の学習コスト
    	- 他の技術スタックを用いたマイクロサービスの切り出しが容易か

- 結果
	- チーム内でコードの責務に関する話題や、アーキテクチャの是非を議論する習慣を形成できました。
	- リファクタリング作業が計画的にタスクとして起票され、実装される習慣を形成できました。

#### サービスの課題と自分が意識した事
##### ETL処理が差分更新に対応しておらず、かつ時間がかかる
- 課題
  - 分析に用いるデータソースの更新を日時のバッチで実行している
    - 1回の実行時間が30分程度かかり、デバッグが非効率
    - 差分更新に対応していないため、一部の分析ユースケースに対応していない
- 取り組み
  - GCPの新サービスである datastream for BigQuery の技術調査と動作検証を実施しました
    - 現状、国内で実務導入されている例はほぼありません
    - Terraform を用いた動作検証を行い、[記事として投稿](https://qiita.com/baeyear/items/f1ce46973511ad999f9a)しました
  - 現在、実務導入に向けて見積もり中です
- 結果
  - モダンなスタックを実務導入する意思決定に対して、一次情報を元にサポートすることができました
  - 実務導入が完了すれば、ETL処理に必要な監視コスト、運用コストがほぼ0になります
